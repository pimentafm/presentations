SHEET 02 / 23
O problema

AI sem estrutura gera caos

Pedidos soltos à AI (prompts), sem histórico do projeto, sem checagens entre quem escreve e quem revisa.

01 · MEMÓRIA
Sem histórico do projeto
Cada conversa recomeça do zero — nada salvo em arquivo, nada para conferir depois.
02 · VOCABULÁRIO
Termos inventados
A AI cria nomes de APIs e conceitos do negócio que não existem — hallucination.
03 · ETAPAS
Planejar vira codificar
Sem gates entre discovery, planning e implementação — tudo misturado.
04 · PAPÉIS
Mesmo agent faz tudo
Quem escreve também revisa e aprova — erros antigos voltam sem ninguém perceber.
A resposta não é só um modelo melhor — é um harness com método e conhecimento RULES · SKILLS · GRAPH · GATES
CONCEITO · O MODELO
SHEET 03 / 23
Fundação

O modelo: o motor

O modelo (LLM: Claude, GPT, Gemini…) é o “cérebro” — prevê texto e raciocina sobre linguagem.

01 · PODER
Raciocínio e linguagem
Resolve problemas, resume, escreve código, explica conceitos.
02 · SEM MEMÓRIA
Esquece entre sessões
Nada fica gravado — cada conversa no chat recomeça do zero.
03 · SEM AÇÃO
Não age sozinho
Sozinho, não abre arquivos, não executa comandos, não consulta sistemas.
04 · SEM GARANTIA
Pode alucinar
Inventa fatos, nomes e integrações como se fossem reais.

Um motor potente — mas sozinho, sem chassi, direção ou freios, não leva ninguém a lugar nenhum.

Trocar de modelo aumenta o poder — não as garantias de qualidadeMODELO = LARGE LANGUAGE MODEL (LLM)
CONCEITO · O HARNESS
SHEET 04 / 23
Fundação

O harness: do motor ao carro

Harness = tudo ao redor do modelo para ele agir com segurança e propósito — como chassi, direção e freios no carro.

PeçaO que faz (em português claro)
Contexto→O que a AI enxerga: instruções, arquivos e dados relevantes
Ferramentas→Permite ler/escrever arquivos, rodar comandos e buscar informação
Memória / Estado→O que fica salvo entre passos e entre sessões de trabalho
Controle de fluxo→Define etapas, ordem e quando parar ou avançar
Guardrails→Trilhos de segurança: rules, review e gates que barram erro

Agent = Model + Harness

Ex.: code-writer = modelo + rules SDD + skill TDD + ferramentas de arquivo — um agent, um papel.

O harness canaliza o poder do modelo em comportamento confiável e repetívelAGENT = MODEL + HARNESS

Glossário essencial

Termos que voltam ao longo da apresentação — em linguagem direta.

LLM / Modelo
AI que entende e gera texto — ex.: ChatGPT, Claude, Gemini.
Harness
Estrutura ao redor do modelo: rules, memória, ferramentas e etapas.
Agent
Modelo + harness com um objetivo — não é só o chat, é o sistema completo.
Prompt
Pedido ou instrução que você envia à AI.
SDD
Desenvolvimento guiado por especificação — definir e validar o “o quê” antes do código, com gates entre fases e estado salvo em disco.
Pipeline
Sequência fixa de etapas em ordem — a linha de montagem do fluxo (ex.: fases SDD).
Gate
Checagem de qualidade: só avança se passar (GO = segue, NO-GO = para).
Hallucination
Quando a AI inventa informação com aparência de verdade.
Skill
Manual reutilizável de instruções que a AI segue em tarefas parecidas.
MCP
Protocolo que permite à AI usar ferramentas externas (dados, arquivos, APIs).
Rule
Instrução fixa sempre ativa — define limites e comportamento padrão.
Auditable
Dá para rastrear o que foi feito, quando e com base em qual evidência.
Ontology
Dicionário oficial do negócio — nomes e significados acordados para a AI não inventar termos.
Graph
Mapa de como código, documentos e conceitos se conectam no projeto.
TDD
Desenvolvimento orientado a testes — escrever o teste antes do código que o atende.
Catalog
Agents especialistas opcionais — você chama quando precisa de opinião, fora do fluxo fixo.
Workflow
Fluxo de trabalho orquestrado — etapas, ordem e quando avançar ou parar (mesma ideia de pipeline; não é revisão de branch).
Guardrails
Trilhos de segurança do harness — limites fixos que impedem o agent de sair do rumo ou pular etapas.
Discovery
Primeira fase SDD — entender contexto e decidir se a feature vale a pena (GO / NO-GO).
Planning
Segunda fase SDD — spec detalhada e tarefas; só avança se o gate de qualidade passar.

SHEET 06 / 23
Vocabulário do domínio

Domain Ontology

Ontology = dicionário oficial do negócio. Domain = área de conhecimento (ex.: crédito, geoespacial, pagamentos). No Omni, omni project init gera ontology em camadas.

Uma ontologia cataloga conceitos do domínio — significado, relações e nomes aceitos no projeto.

Exemplo · ontology.yaml (domínio crédito)
# .omni/domains/credito/ontology.yaml concepts: bureau_score: label: Bureau Score definition: Pontuação de crédito no bureau reject_synonyms: [credit_score_api] # inventado pela AI
ArtefatoPara que serve
Arquivo de ontology (YAML)→Termos, contextos e sinônimos aceitos pela AI
Glossário humano (Markdown)→Mesmo vocabulário, legível para pessoas
Pacote compacto por feature→Resumo ≤4KB enviado à AI — sem ler o repositório inteiro
menos invenção vocabulário compartilhado lacunas detectadas alertas de inconsistência
Sem ontology, a AI inventa nomes; com ontology, ela consulta o dicionário oficialVOCABULARY

Mapa do projeto

Knowledge Graph

Graph = mapa de como código, documentação e conceitos se conectam. A AI consulta o mapa em vez de abrir dezenas de arquivos ao acaso.

Como se constrói

De código a mapa

  • Análise automática do código → índice estruturado
  • Relações gravadas em banco (módulos, funções, docs)
  • Conceitos do negócio ligados ao código
Ferramentas MCP

Consultas compactas

Busca exata, vizinhos, resumos por feature — e busca semântica opcional (graph_semantic_search) por significado, não só por nome. Clique para ver o fluxo.

Do caos à entrega com evidência

Por que SDD?

Spec-Driven Development — o harness aplicado ao fluxo de entrega: especificar e validar antes de codificar.

01 · PROBLEMA
Fim do vibe coding
Pedido solto vira código sem critério. SDD força: ideia → spec → gate → só então implementar.
02 · ENGENHARIA
Garantias no disco
Estado em .omni/, não no chat. Gates quantitativos. Quem escreve ≠ quem revisa.
03 · NEGÓCIO
Entrega confiável
Previsível, rastreável e auditável — decisões GO/NO-GO registradas antes de investir em código.
Contraste
Sem SDD: prompt → código → surpresa → retrabalho
Com SDD: ideia → spec → validação → código com critério
spec antes de código estado em arquivos papéis separados retomar de onde parou
SDD não é burocracia — é método para o agent não pular etapasPROPÓSITO SDD

SHEET 10 / 23
Controle de fluxo do harness

SDD — Discovery → Planning → TDD → Review

SDD coloca etapas e gates no harness: cada fase só começa se a anterior passar no critério. Progresso salvo em disco — auditable.

  • Discovery — vale a pena fazer? Nota de viabilidade e decisão GO/NO-GO
  • Planning — spec + tarefas; gate exige nota 3/3 por seção
  • TDD — teste antes do código; revisão ≥4/5; garantia de qualidade ≥70
  • Review — revisão opcional de todo o trabalho no branch
Exemplo · pipeline de uma feature
/feature-discovery → decision.json (GO)
/feature-planning → quality-gate.json (3/3)
/feature-tdd → RED → GREEN → review.json
/workflow-reviewer → revisão do branch antes do merge
fases independentes estado em arquivos modo rigoroso retomar de onde parou
Cada fase produz documentos em disco — não depende da memória do chatMÉTODO SDD

SHEET 11 / 23
Especificar

Discovery — vale a pena?

Antes de planejar ou codificar: entender contexto, carregar ontology do domain e avaliar viabilidade.

Agents (especialistas)

Painel de descoberta

  • Analisa contexto do projeto
  • Arquiteto + especialista de stack
  • Orquestrador consolida notas
Gate

GO / CONDITIONAL / NO-GO

GO = ≥18/25 (todas seções ≥2). CONDITIONAL = 14–17. NO-GO = <14. Vocabulário carregado antes de ler o código.

Exemplo · decision.json
{ "decision": "GO", "total_score": 18, "section_scores": { "problem_clarity": 4, "feasibility": 4, "blast_radius": 3, "risk": 4, "effort": 3 } } // 18 ≥ 18 → GO · gravado em .omni/features/<id>/discovery/
Resultado: decisão registrada em arquivo — auditable e retomávelFASE 1 · DISCOVERY

SHEET 12 / 23
Planejar

Planning — quebrar em partes

Transformar a ideia em especificação escrita, tarefas pequenas e critérios claros de “pronto”.

  • spec.md — documento com o que será feito e por quê
  • tasks.json — até 5 tarefas, ≤4h cada, ≤5 critérios de aceite por tarefa
  • quality-gate.json — cada seção da spec precisa de nota 3/3
  • Painel crítico: requisitos, QA e segurança revisam antes de codificar
Exemplo · quality-gate.json
{ "sections": { "requirements": 3, "architecture": 3, "tasks": 2, "security": 3 }, "passed": false, "threshold": "each_section_gte_3" } // tasks=2 → NO-GO · volta para planning, não entra em TDD
Agents escolhidos conforme o domain do projeto (ex.: web, dados, infra)FASE 2 · PLANNING

SHEET 13 / 23
Implementar

TDD — teste antes do código

TDD (desenvolvimento orientado a testes): 1) escrever teste que falha 2) código que passa 3) melhorar — com quem escreve ≠ quem revisa.

RED → GREEN → REFACTOR
→ code-reviewer (nota ≥4/5) → assurance (≥70)
Exemplo · review.json (notas do gate TDD)
{ "reviewer": "code-reviewer", "sections": { "correctness": 5, "tests": 4, "security": 5 }, "passed": true, "min_threshold": 4, "notes": "Writer separado do reviewer; testes cobrem AC da tarefa." }
  • Só o code-writer agent altera o source code
  • Tarefas em paralelo quando não dependem uma da outra (até 5 por vez)
  • Branch de trabalho isolada, criada a partir da branch principal
Assurance: um agent avalia, outro critica — nota mínima para aprovarFASE 3 · TDD

Quem faz o quê

Pipeline vs Catalog

Pipeline = agents obrigatórios do fluxo SDD. Catalog = especialistas opcionais que você chama quando precisa de opinião.

22 · Pipeline

Gates obrigatórios

Orquestram SDD, testes, assurance e revisores por tecnologia. Não podem ser substituídos por catalog.

  • orchestrator — aplica gates entre fases
  • code-writer — único que altera código
  • code-reviewer — nota ≥4/5 por tarefa
200 · Catalog

Consultoria sob demanda

Especialistas por área (marketing, finanças, segurança…) — instalados quando necessário. Só leitura por padrão.

  • security-reviewer — opinião AppSec
  • proposal-strategist — RFP / proposta
  • fora do fluxo SDD obrigatório

Decisão prática

Quando usar cada peça

Cada componente do harness entra em um momento diferente — resumo abaixo.

PeçaQuando usar
Rule→Sempre. Limites fixos — git, SDD, segurança. Não negocia por prompt.
Skill→Tarefa repetível. Passo a passo (TDD, commit) — carrega sob demanda.
Agent→Papel especializado. Writer ≠ reviewer. Catalog = advisory.
Pipeline→Entregar com gates. Feature ou bug — discovery → planning → TDD.
Router→Chat livre. Graph-first → catalog advisory. SDD bypassa.
Workflow→Fechar o branch. Pós-TDD, pré-merge — /workflow-reviewer.
Rule = guardrail Skill = receita Agent = papel Pipeline = linha SDD Router = chat livre Workflow = review
Regra de ouro: catalog aconselha — pipeline e workflow garantem entregaGUIA DE USO
